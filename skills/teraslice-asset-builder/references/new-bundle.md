# Scaffolding a New Asset Bundle Repo

An asset bundle is a repo (usually a small monorepo) that compiles to a deployable zip. Model a new one on `standard-assets` or `elasticsearch-assets` in `/Users/jsoto/Workspace/TerasliceAssets/`. Confirm the target directory and bundle name with the user first.

> **Talking to an external service (database, queue, object store)?** The bundle must also ship, **inside its own monorepo** (never in the `teraslice` repo):
> - a **terafoundation connector** (`packages/terafoundation_<name>_connector`) — **required when introducing a new service such as a database**; it holds the connection config (host/user/password) so credentials stay out of the job, which references a connection by name via `_connection`;
> - a companion **API/client package** (`packages/<name>-asset-api`);
> - a **Dockerfile** that installs the connector onto the Teraslice runtime image.
>
> Read `references/api-package-and-docker.md` for all three — it is a required part of this kind of bundle.

## Minimum layout

```
<bundle-name>/
├── package.json            # root: build/test/lint scripts, devDependencies
├── tsconfig.json
├── pnpm-workspace.yaml      # if using workspaces (asset + packages)
├── asset/
│   ├── asset.json          # { name, version, minimum_teraslice_version }
│   ├── package.json        # runtime dependencies of the operations
│   ├── tsconfig.json
│   └── src/
│       ├── index.ts        # the registry (auto-generated)
│       └── <operation>/    # first operation
│           ├── processor.ts (or fetcher.ts + slicer.ts)
│           ├── schema.ts
│           └── interfaces.ts
└── test/
    └── <operation>/
        ├── processor-spec.ts
        └── schema-spec.ts
```

At deploy time the runtime expects a `package.json`, an operations directory, and the `node_modules` the ops need. Keep `devDependencies` and test files at the **root**, not inside `asset/`, so they're excluded from the bundle zip.

## `asset/asset.json`

```json
{
    "name": "my-bundle",
    "version": "0.1.0",
    "minimum_teraslice_version": "3.3.0"
}
```

Optional fields restrict where the asset can run: `node_version`, `platform`, `arch`. Omit or leave falsy to keep the bundle portable across node versions and OSes. `description` is commonly added too.

## `asset/package.json`

- `"type": "module"` — bundles are ESM.
- Depend on `@terascope/job-components` (base classes) and `@terascope/types` (`OpConfig`), plus whatever clients the ops use.
- A `build` script: `"build": "tsc --project tsconfig.json"`.

## Root `package.json`

Mirror an existing bundle. Typical scripts (they wrap `@terascope/scripts` = `ts-scripts`):

```json
{
  "scripts": {
    "setup": "pnpm install && pnpm build --force",
    "build": "tsc --build",
    "build:watch": "tsc --build --watch",
    "test": "ts-scripts test asset -- --testPathIgnorePatterns=test/e2e",
    "lint": "eslint",
    "lint:fix": "pnpm lint --fix"
  }
}
```

Root `devDependencies` usually include `@terascope/scripts`, `@terascope/eslint-config`, `teraslice-test-harness`, and `@jest/globals`.

## The registry — `asset/src/index.ts`

Auto-generated; maps each op name to its classes. Generate/refresh it with:

```bash
teraslice-cli assets registry     # or: assets init
```

Shape:

```ts
// NOTE: auto-generated by `teraslice-cli assets registry` — manual edits are lost on regen.
import MyOp from '../src/my_operation/processor';
import MyOpSchema from '../src/my_operation/schema';

export const ASSETS = {
    my_operation: {
        Processor: MyOp,
        Schema: MyOpSchema,
    },
};
```

For a reader the entry uses `Fetcher` + `Slicer` + `Schema`; for an API it uses `API` + `Schema`.

## Build & deploy

```bash
pnpm run setup                          # install + build
teraslice-cli assets build              # produces build/<name>-vX.Y.Z-node-<v>-bundle.zip
teraslice-cli assets deploy <cluster> --build
```

## Using the bundle in a job

List the bundle name in `assets` and call operations by their `_op` name:

```json
{
    "name": "example",
    "lifecycle": "once",
    "workers": 1,
    "assets": ["my-bundle"],
    "operations": [
        { "_op": "my_reader" },
        { "_op": "my_operation", "field": "foo", "value": "bar" }
    ]
}
```
